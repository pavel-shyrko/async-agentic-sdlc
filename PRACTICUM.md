# Engineering Practicum: Project Manifest

An engineered, deterministic multi-agent workflow engine that automates the full SDLC
without human intervention. This document is the high-level manifest and the distilled
engineering record. The chronological development history — release-by-release — now
lives in [CHANGELOG.md](./CHANGELOG.md); the decisions behind each iteration are recorded
as Architecture Decision Records in [docs/decisions/](./docs/decisions/).

## Key Engineering Takeaways
* **To Loop a One-Shot Pipeline, First Make Its Halt a Catchable Event — Separate "the Run Halted" from "the Process Must Exit"** *([ADR 0019](./docs/decisions/0019-cyclical-multi-ticket-orchestration.md))*: Turning a single-shot pipeline into a batch that drives *every* unit of work to completion looks like "add a `for` loop," but the real blocker is usually the termination primitive: the halt path calls `sys.exit(1)` directly, so the **first** unit's failure kills the whole process — no chance to record what already succeeded, no resume. The durable move is to replace the bare exit with a **catchable domain exception** (`PipelineHalt`) raised *after* the halt's side-effects (incident report, FinOps) are persisted, and push the exit-code decision to the **entrypoint** (`main.py`: `except PipelineHalt → sys.exit(1)`). Now a single-unit run exits exactly as before (behaviour-identical, its tests prove it), while the batch loop *catches* the halt to checkpoint which units finished and stop cleanly. Three design choices make the loop trustworthy. (1) **Persist batch progress in its own checkpoint** (`BatchState` → `reports/batch_state.json`, beside the planning checkpoint that the resume verb already resolves) so a bare `--resume <project>` re-enters the batch, **skips completed units, and re-runs the failed one fresh** against the now-current shared state — resuming its stale per-unit checkpoint would rebuild against an out-of-date world. (2) **Keep only FSM halts catchable**: infra `sys.exit(1)`s (clone/push/merge) stay plain `SystemExit` so the batch's `except PipelineHalt` deliberately does *not* swallow them — they kill the process and resume recovers. (3) **Encode the coupling the work-units actually have**: because each unit clones `main` fresh, unit N+1 sees unit N's output *only if it merged first*, so the batch flag (`--auto-execute`) **implies** the merge flag (`--auto-merge`) — an unmerged batch is incoherent, so the CLI makes the incoherent mode unrepresentable. The cost accepted, stated not hidden: the financial breaker is still **per-unit** (each `run_executor` measures only its own spend), so a batch can overspend `N×` the intended ceiling until an application-wide budget threads the remaining limit cycle-to-cycle (deferred as epic E5).
* **Close an Autonomy Loop's Last Mile Behind a Provider-Agnostic Seam — and Decide the Identity Model Before You Build It** *([ADR 0018](./docs/decisions/0018-auto-merge-pr-loop-closure.md))*: The final step of an autonomous pipeline — landing verified work in `main` — is an *integration* decision, not a code-generation one, and the trap is to inline the forge's CLI (`gh pr create/merge`) straight into the success block. The durable move is a **three-function seam** (`open_pr`/`approve_pr`/`merge_pr`) in the shared plane, so the provider (GitHub today, GitLab/Bitbucket later) is swappable and the call sites never parse URLs — the forge infers owner/repo from the clone's `origin` remote. Two non-obvious constraints must be resolved **up front**, not discovered in production. (1) A forge forbids a PR author approving their *own* PR, so "auto-approve" is not one credential but **two identities** (a separate reviewer token) or an admin bypass — name which you use, and make approval *best-effort* (log-and-swallow any failure) so a missing/invalid reviewer token never aborts the merge the admin path can still complete. (2) **Local gates are not remote gates**: an immediate admin merge can be refused for *pending required checks*, so detect that stderr signal and fall back to a queued (`--auto`) merge instead of failing. Make the step **idempotent on resume** (reuse an OPEN PR into the same base, skip a MERGED one) because it runs after a push that may already have half-happened, and keep the bridge in the entry/worker layer so the control plane never imports the worker just to open a PR (ADR 0012 holds). The cost accepted, stated not hidden: a real approval means *operating* a second identity, and a merge made with a CI bot's own token will not cascade downstream workflows (that needs a real PAT).
* **Defend a Boundary at the Boundary — Agent-Authored Text Is Untrusted Bytes, and a Blocking Call Without a Ceiling Is an Unbounded Hang** *([ADR 0018](./docs/decisions/0018-auto-merge-pr-loop-closure.md))*: Two separate production stalls in one iteration shared a single root shape — an unguarded value reaching a system boundary — and both are fixed *at the boundary*, as a shared SSOT covering every call site of that kind, not by patching the one field that happened to break. (1) **Treat model/agent output as untrusted bytes.** A corrupted glyph (a `©` mangled to `\x00`) in a generated ticket flowed into a PR body and crashed `gh pr create`, because POSIX `execvp` rejects a NUL in *any* argv element. The fix is one sanitizer applied at *every* subprocess-argv seam — the new `gh` path **and** the pre-existing `git commit` path, which carried the identical latent bug — stripping control chars while preserving legitimate `\t`/`\n`/`\r`. (2) **Bound blocking calls at the transport, not with a cancel-wrapper.** A stalled structured-LLM request hung the engine *forever*, because the retry layer only catches *exceptions* (a silent network stall is not one) and the call ran on a thread pool task that cannot be cancelled — so an `asyncio.wait_for` wrapper would merely orphan the worker thread and leak the pool. The durable fix is a **timeout on the client/HTTP layer itself** (`http_options.timeout`), so a stall actually *raises* → the existing backoff fires → it fails fast. The generalizable rule: marshal and time-bound at the edge (argv encoding, transport ceilings) as a single SSOT for all crossings of that boundary — the next corrupted glyph or slow endpoint will arrive through a different field or call than the one you just patched.
* **A Feature That Chains Two Phases Is First a Refactor: Extract the Callable Seam Before Wiring the Chain** *([ADR 0017](./docs/decisions/0017-nexus-executor-auto-dispatch.md))*: When two phases of a workflow (here: plan, then execute) can only be run as two separate operator commands, the temptation is to bolt a "do both" flag onto the entrypoint. But the real blocker is usually structural — the second phase's logic was **inlined inside `main()`** (a ~350-line FSM body), not a callable — so "chain them" actually means "extract the inlined body into a function, *then* call it twice." Do the extraction as a **pure, verbatim lift** (the moved body took no new parameters; the only behavioral change was `return` → `return True`) so the existing path stays byte-identical and its tests prove it; the new capability is then a thin caller, and — the larger payoff — the extracted function (`run_executor`) becomes the reusable seam every *future* step (iterate all tickets, a failure policy) builds on, instead of each one re-cutting the same ~350 lines. Two boundary lessons rode along. (1) **Plumb the chaining at the entry layer, never by letting the planner import the worker** — the dispatch lives in `main()` and the executor imports the task-list helper from the planner lazily, so the control plane still never depends on the worker plane (the ADR 0012 import discipline survives the new coupling). (2) **"Nothing to do" is not "it failed": make exit codes encode that** — once planning *succeeds*, every downstream skip (no repo, no tickets, missing ticket file) is a clean **exit-0** warning, and the process exits **1** only on a genuine halt *inside* the executor; conflating an empty work-list with an error would make every harmless skip look like a broken run. The cost accepted: only the *first* ticket auto-runs, and a halt still `sys.exit(1)`s the whole process — correct for one ticket, but a precondition (non-exiting halts) that must be paid down before multi-ticket dispatch is safe, so the seam was drawn deliberately wider than the feature that first used it.
* **Treat Architecture Docs as a Drift-Prone Artifact with an Enforced Sync Path** *([Iteration 16.1](./docs/releases/iteration_16.1/iteration_16.1_README.md))*: Hand-authored architecture documentation — a C4 diagram, a component table, an onboarding guide — decays exactly like a stale README roster: the moment a new agent role, FSM route, or container lands, the picture silently lies, and a one-off "we wrote the docs" effort guarantees it. The durable move is not to write better docs once but to make each doc a **maintained target of an existing sync mechanism**: the same skill that already reconciles CHANGELOG/README gained an *Architecture Diagram Sync* step with a structural trigger (new/removed role, route, external system, or plane) and a grep-the-peer drift sweep, so the diagram and the component table re-sync on the same cadence as the code. Two corollaries proved their weight. (1) **An onboarding guide must mirror what the program actually enforces, not a tidy ideal** — the setup guide's pre-flight self-check was written to reproduce `check_environment()`'s exact gate (`docker`/`claude`/`bandit` on PATH + `GEMINI_API_KEY`), so a reader passes the engine's check on the first run instead of discovering it as a `🚨 CRITICAL` exit; a guide that lists aspirational steps the binary doesn't verify is worse than none. (2) **License the tool and its output as separate decisions** — the engine adopted Apache-2.0 (explicit patent grant + change-notice, fitting a code-generating system) while the MIT boilerplate it injects into *generated apps* stayed MIT; conflating "what license does this repo have" with "what license do its artifacts carry" is a category error. The cost accepted: docs sync is still prompt/skill-governed (a structural change with no matching skill trigger can still slip), so the trigger list itself is now load-bearing and must grow with the architecture.
* **Give a Self-Healing Loop a Route to Repair the Spec, Not Just the Workers** *([ADR 0016](./docs/decisions/0016-arbiter-contract-self-healing.md))*: A retry loop that can only route a failure to the agents that *implement* the work (here: Developer for code, QA for tests) silently assumes every failure is implementation-fixable. When the defect is actually in the **contract/spec** — a mandated algorithm that is self-contradictory, overlapping error conditions with no declared precedence, or a "fix" that could only satisfy a test by violating a stated NFR — no downstream agent can resolve it, so the loop reproduces the same failure every cycle until the breaker. The defect is invisible precisely because "the spec is wrong" is not a representable outcome. The generalizable fix is a **triage agent that adds a third route — back to the spec**: classify the stuck failure, and when it is contract-level, re-derive (amend) the contract instead of re-prompting a worker. Make this *safe*, not just *powerful*: **pin the highest-cascade field** (the platform/`environment_id` that selects sandbox, gates, and test layout) so an amendment can never thrash it; **cap autonomous rewrites** (default 1, else halt-to-human) to bound spec-thrash; **reuse the existing re-validation machinery** (regenerate tests, re-run guardrails) rather than inventing a parallel one; and grant a bounded **retry-budget bonus** per amendment so the re-derived spec gets a fair shot, with the financial breaker still the absolute ceiling. Pair it with upfront prompt hardening (declare error precedence; forbid validators that short-circuit before a more specific error surfaces) so the *first* contract is usually right and the amendment route is a rare safety net — observed end-to-end, the triage agent most often correctly *declines* to amend and lets the cheaper worker channel converge. The cost accepted: the contract becomes mutable mid-run (so a checkpoint before vs. after an amendment describes a different spec — disambiguated by an amendment counter), and the triage classification is itself prompt-governed, bounded but not eliminated by the route enum + the amendment cap.
* **Don't Retry a Deterministic Block — Remove Its Trigger at the Source** *([ADR 0016](./docs/decisions/0016-arbiter-contract-self-healing.md))*: A content-filter refusal (Gemini `RECITATION`/`SAFETY`/…) is **deterministic** — the identical prompt yields the identical block — so a generic exponential-backoff retry layer burns its whole budget reproducing the same failure, then halts anyway. Two compounding lessons. First, **classify non-retryable failures and fail fast**: a retry policy must distinguish transient faults (network, 5xx) from deterministic refusals, and for the latter either stop immediately or take a *different* action — here a single paraphrase-guarded retry that follows the error's own hint ("reduce verbatim quoting") rather than re-sending the blocked prompt. Second, and more durable, **fix the root cause by removing the trigger, not by coping with the symptom**: the block was *caused* by a prompt instructing the model to reproduce canonical, training-data-matching boilerplate (a full MIT `LICENSE`, a verbatim `.gitignore`). The right fix is to stop asking a probabilistic model to regurgitate text the *engine already knows* — inject that canonical content deterministically and let the model produce only the novel, project-specific parts. The generalizable rule: never route deterministic, engine-knowable content through the LLM; reserve generation for what genuinely requires reasoning.
* **Unify Subsystems by Location SSOT + a Self-Describing Checkpoint, Not Shared State** *([ADR 0015](./docs/decisions/0015-unified-project-run-topology.md))*: When two planes of one system (control-plane planning and worker-plane execution) each invented their own run location — one UUID-keyed under `runs/`, the other a hardcoded directory entirely outside it — there was no shared lineage, no aggregation, and resume understood only one of them. The instinct is to merge their *state*; the cheaper and more durable move is to merge only their *location and resume envelope*, and keep the state models separate. Draw one filesystem SSOT (`Projects` + `allocate_run_dir`) that names every run `<project>/<NNN>_<plane>_<label>_<ts>_<uid>` so a glance reveals the task, the plane, and the order — human-readable slugs and sequential numbers in place of opaque UUIDs, with a per-project `project.json` umbrella captured once and reused. Then give the laggard plane the *same* checkpoint dump/load contract the other already had, and make the checkpoint **self-describing**: a `kind` discriminator field lets `--resume` route to the correct plane by reading the JSON, with zero directory-name parsing — so the two state models (`NexusState` vs `GlobalPipelineContext`) stay deliberately distinct, bridged only by a tagged envelope. Two design corollaries pay off: inject the runs-base (`Projects(base)`) instead of reading a module global, so the whole layer is hermetic against a temp dir under test; and recompute per-run meta-dirs (`logs/`/`reports/`/`artifacts/`) from `run_dir` rather than persisting them, so a relocated or copied run never carries stale absolute paths. The cost accepted: the run-dir *name* becomes load-bearing metadata whose grammar must stay stable, and slug collisions resolve by suffixing — which can silently detach a later run from the umbrella a user expected.
* **Don't Make a Probabilistic Model Do Deterministic File Surgery** *([ADR 0013](./docs/decisions/0013-structured-test-maintenance-ast-pruning.md))*: Asking an LLM to "preserve the existing test file and append new cases" makes a stochastic process responsible for a deterministic operation — preserve N, add M, remove K — so it loses cases, burns output tokens re-emitting the whole file, and risks truncation/hallucination; bolting on a code-side guard that spills to `_v2.py` zombie files only trades destruction for fragmentation. The generalizable fix is a strict labour split: the model returns only the **deltas** it is good at producing (new code, new imports, and the *names* of now-obsolete cases), and a deterministic engine does the surgery — here, parse the file to an AST, drop the obsolete top-level nodes by name, `ast.unparse`, dedupe imports, and append the new code in place. Preservation becomes a guarantee of the engine, not the goodwill of the model. The cost is that the engine must be hardened against the mechanical failure modes the model used to paper over (duplicate imports, concatenation `SyntaxError` from lost separators, a misplaced `__main__` guard, non-utf8 locales, an unparseable on-disk file), and AST round-tripping normalises away comments/formatting — an acceptable price for machine-generated artifacts.
* **Virtual Separation Before Physical Separation** *([ADR 0012](./docs/decisions/0012-virtual-separation-monorepo-planes.md))*: A flat namespace that mixes the control logic (what work to schedule/deploy), the worker logic (one execution run), and the shared foundations makes the eventual cost of splitting them unbounded — every consumer must be re-traced by hand to find the seam. The generalizable move is to draw the boundary *logically first*: establish the planes as packages (`nexus`/`executor`/`shared`) enforced purely by import discipline, with a thin entrypoint that decouples "where the program starts" from "where a run executes", **before** paying for any physical split (separate services, repos, or dependency sets). This is a pure lift-and-shift — history-preserving `git mv` plus a repo-wide import rewrite, zero logic change — so behaviour is provably identical (the full suite stays green) while a future extraction becomes a mechanical cut along an already-drawn line. The one hazard to watch: `__file__`-relative path anchors silently break when a module changes directory depth and must be re-based as part of the move (here, `prompts.py`'s `_REPO_ROOT` needed one extra `.parent` to keep resolving the repo-root `prompts/` tree).
* **Language-Neutral Contracts Decouple Architecture from Stack** *([ADR 0011](./docs/decisions/0011-secure-sandbox-and-finops-telemetry.md))*: When a design contract specifies how modules wire together as concrete import statements (`from pkg import X`), it silently couples the *architecture* to one language's syntax and leaks that syntax to every downstream agent — the same design can no longer drive a `dotnet` or `typescript` target. The fix is to make the design node the Single Source of Truth for *structure* but never *syntax*: emit a language-neutral dependency **graph** (`file_path`, `exports`, and `depends_on` links as `path:symbol`, no import statements), declare the target language separately as a routing tag, and let the execution agents translate neutral links into the target language's imports. This is an Open-Closed boundary — adding a language changes only the routed syntax skill, never the contract — at the cost of one more well-formed surface the design node must populate correctly.
* **Secure WSL Sandboxing & Real-Time FinOps Circuit Breakers** *([ADR 0011](./docs/decisions/0011-secure-sandbox-and-finops-telemetry.md))*: A developer-convenience daemon bound to `0.0.0.0:2375` without TLS is not a convenience but a host-level RCE — the plaintext Docker API, published on every interface, hands any subnet process root on the host via a privileged bind mount; sandbox network surfaces must therefore default to loopback (`127.0.0.1`), treating remote reachability as an explicit, separately-secured opt-in rather than the baseline. Symmetrically, cost telemetry that is only available *retrospectively* (an out-of-band `ccusage` report) cannot gate an autonomous retry loop: by the time the bill is read, the budget is already spent. The generalizable rule is that spend must be a first-class, real-time FSM signal with its own deterministic hard-halt — a Financial Circuit Breaker that bounds *cost* exactly as the functional breaker bounds *attempts*. Money math feeding that gate must be exact: pricing is computed in `Decimal`, since IEEE 754 floats drift on fractional-cent rates and would trip the budget threshold early or late by an accumulating margin. (The role that authors these gated contracts is the `TechLead` — renamed from `Architect` to name the binding, machine-readable contract it actually emits.)
* **Fast-Fail Middleware & Legacy Protection** *([ADR 0010](./docs/decisions/0010-fast-fail-documentation-guardrail.md))*: Handing a probabilistic LLM gate a blunt, binary enforcement directive ("delete anything outside the contract") destroys legacy state: the Reviewer cannot distinguish a mangled pre-existing file or a justified new helper from hallucinated garbage, so each eradication breaks imports and burns the functional retry budget in a token-draining loop. The generalized fix is two-layered — a deterministic, zero-token lexical middleware between agent phases that fast-fails cheap violations (an undocumented new file) straight back to the producer without touching the expensive gate nodes or the circuit-breaker budget, plus a graduated triage policy at the LLM gate (justified addition / hallucinated garbage / legacy victim → revert, never eradicate) instead of binary enforcement. Capping the local reroutes (MAX=2) with a deterministic Hard Halt that dumps full FSM state to an `incident_report.json` converts a previously unbounded failure loop into a bounded, auditable incident.

* **Hybrid Skill Routing & OCP** *([ADR 0009](./docs/decisions/0009-hybrid-skill-routing.md))*: Hardcoding agent instructions — even when the rules live in separate files, if each node still hand-lists which to load — violates the Open-Closed Principle and leads to context pollution: broad rules get copy-pasted across nodes and narrow rules bleed into agents they do not apply to. The fix is declarative tagging: each skill file carries `type`/`nodes`/`triggers` frontmatter, and a single router (`build_agent_context`) composes the per-node skill set, gating `domain` skills on a tag intersection with the contract. Combined with a lightweight, dependency-free LLM semantic fallback (a structured relevance check that catches relevant skills whose author forgot a trigger tag), this enables effectively infinite horizontal scaling of domain expertise — a new guardrail is a tagged markdown file, never a code change to the core orchestrator. The trade-off to accept consciously: targeting moves from code to metadata, so it fails *open* (a malformed tag silently drops the skill) and a tag miss costs a probabilistic extra LLM call.

* **Stateful FSM Isolation & Component Skills** *([ADR 0008](./docs/decisions/0008-git-anchored-sessions-atomic-commit.md))*: Stateless retries cause FSM Catch-22 deadlocks: when a failing QA cycle re-enters the Developer node without injecting the `test_code_snapshot` from that cycle, the Developer regenerates from the same broken baseline, burning the entire retry budget without forward progress. The fix is mandatory state injection — `--reset-attempts` preserves `test_code_snapshot` while clearing `current_attempt`, and sequential LLM calls enforce causal order (QA output always precedes the next Developer call). Symmetrically, monolithic inline prompts cause context leakage: a single system string that encodes all agent rules floods unrelated nodes with irrelevant guardrails and makes any rule change a code deployment. The fix is an atomic Skill System: each `prompts/skills/*.md` file encodes one rule; `get_skill` composes only the skills relevant to the current node, making guardrails independently editable without touching application code.

* **The Git Index as a Transactional Unit-of-Work** *([ADR 0008](./docs/decisions/0008-git-anchored-sessions-atomic-commit.md))*: When self-healing agents rewrite files across multiple retry cycles, committing per cycle litters the branch with throwaway intermediate commits and couples the snapshot to branch names. The fix is to treat the cloned repo's Git **index** as the unit-of-work: agents only ever `git add -A` into the index (snapshots are an index diff, `git diff --cached <base>`), while the orchestrator performs a **single atomic commit** on the `feat/ticket-<id>` branch ONLY after every gate passes — guarded against empty commits via `git diff --cached --quiet`. This decouples in-cycle mutation from durable history and makes the causal delta branch-name-independent. The corollary is that the workspace must be anchored to a real clone (resolved via `git rev-parse --show-toplevel`, never a guessed parent path), so the engine operates on arbitrary repositories rather than a hardcoded sandbox.

* **Prompt Engineering Belongs in Prompts, Not in Schema Definitions** *([ADR 0007](./docs/decisions/0007-prompt-schema-layer-separation.md))*: When behavioral LLM directives live inside Pydantic `Field(description=...)` strings, tuning an agent requires editing Python infrastructure code and redeploying the core package — and the same rule exists in two places (schema + system prompt) with no enforcement of consistency. The fix is a strict layer boundary: schema `Field` descriptions carry only dry structural intent ("Boolean flag indicating approval status"); all behavioral rules ("Set to `true` ONLY IF tests are written without loopholes") migrate to a `### Output JSON Schema Semantics` section in the agent's system prompt file, bound per JSON key. An `lru_cache`d prompt loader resolves markdown files from `prompts/system/` at runtime, making behavior editable without touching application code. Iteration-7 validation across the three reference contracts (Prime 1 cycle / 32 tests, Fibonacci 1 cycle / 16 tests, Geometry 3 cycles / 70 tests, all zero-SAST) found **zero system-prompt leakage into the sandbox** (`artifacts/code/`), confirming the boundary holds at runtime. The generalizable rule: infrastructure schemas describe shape; system prompts describe judgment.

* **Transactional Integrity Across Ephemeral Runtimes** *([ADR 0006](./docs/decisions/0006-fsm-state-serialization-resume.md))*: Any pipeline whose state lives only in a Python process is one `Ctrl+C` away from full token loss. The fix is a single rolling `checkpoint.json` written by `GlobalPipelineContext.save_checkpoint` after every critical node, plus a `--resume` CLI path that loads it through `model_validate_json` and **bypasses** any node whose output already exists in the restored context. The Circuit Breaker counter (`current_attempt`) must be promoted from a loop variable to a persisted Pydantic field — otherwise process restarts silently reset the retry budget and defeat the safety guarantee. A companion `--reset-attempts` flag covers the canonical "I fixed the prompt, retry from scratch but keep the contract" workflow. The generalizable rule: every piece of ephemeral runtime state that gates routing decisions (e.g. `regenerate_tests`) must be **derivable from persisted fields** on resume — never reconstructed from in-memory flags.

* **Confirmation Bias as a Token Tax** *([ADR 0006](./docs/decisions/0006-fsm-state-serialization-resume.md))*: When the Developer agent (Claude CLI) was free to "helpfully" emit unit tests next to production modules, those tests polluted the `production_code_snapshot` and effectively let the Developer grade its own work — a textbook confirmation-bias loop that wasted QA tokens regenerating already-passing assertions. Symmetrically, QA tests that asserted on exception strings (`assertIn("must be positive", str(exc))`) tightly coupled the test suite to the Developer's wording, producing false negatives on any prompt rewording. Resolution: hermetic prompt-level prohibitions — Developer is explicitly forbidden from writing tests; QA is explicitly forbidden from inspecting exception strings. Scope isolation must be enforced at the LLM boundary, not by post-hoc filtering of agent output.

* **Architectural Invariants Belong in the TechLead Prompt** *([ADR 0006](./docs/decisions/0006-fsm-state-serialization-resume.md))*: Tight Coupling (e.g. `Cylinder` instantiating `Circle()` inside its body) is an architectural defect that the Reviewer can detect only after tokens are spent on the wrong code. Encoding the invariant — "Classes must receive their dependencies via the constructor; STRICTLY FORBIDDEN from instantiating dependencies internally" — directly into the TechLead's system prompt blocks the antipattern at the contract layer. Per-agent prompts are the cheapest enforcement boundary for cross-cutting design rules.

* **Git Anchor as a State Machine** *([ADR 0005](./docs/decisions/0005-git-driven-state-tracking-qa-fanout.md))*: Naive file-glob scanning of agent sandboxes breaks on real projects because Python runtimes silently write binary `.pyc` files, prior-cycle artifacts bleed into the next snapshot, and there is no scope boundary between "files the agent touched" and "files that happened to exist." Replacing glob scans with a two-primitive Git API (`init_sandbox_git` + `get_pipeline_snapshot_files`) solves all three: `.gitignore` blocks binary noise before it reaches the diff, `git diff <base_branch> --name-only` produces a strict causal delta, and the base-branch anchor is immovable so retries never accumulate stale state. This is a transferable pattern for any agentic system that writes files.

* **LLM Output Fan-Out to Break Token Ceiling** *([ADR 0005](./docs/decisions/0005-git-driven-state-tracking-qa-fanout.md))*: Monolithic LLM generation of a test suite covering an entire multi-file contract hits the `max_output_tokens` wall and returns a truncated, syntactically broken file. The fix is a fan-out: one isolated LLM call per production module, results collected via `asyncio.gather`, each writing its own `test_<module_slug>.py`. This trades one large serial call for N small parallel calls — latency is bounded by the slowest module, total output tokens per call shrink to a single module's scope, and test isolation improves as a side effect. Facade modules (`__init__.py`) must be filtered before fan-out to prevent exponential coverage duplication.

* **FSM Catch-22: TechLead Contract vs. Developer SOLID Compliance** *([ADR 0005](./docs/decisions/0005-git-driven-state-tracking-qa-fanout.md))*: A deadlock emerges when the Developer agent (following DRY/SOLID) extracts shared logic into a utility file not listed in the architecture contract, and the Reviewer agent rejects this as a "contract violation." Neither agent can resolve the conflict within the current FSM cycle — the TechLead cannot amend the contract retroactively, and the Developer cannot remove the helper without violating the design constraints. Resolution requires an explicit authorization rule in the Reviewer's system prompt: utility files created to enforce separation of concerns are pre-approved. This teaches a general principle: FSM gate agents must encode an explicit policy for authorized deviations, not just enforcement of the original contract.

* **The Python Type-Hierarchy Trap & Self-Healing** *([ADR 0004](./docs/decisions/0004-modularization-sandbox-hardening.md))*: Python's type system implicitly inherits `bool` from `int` (`isinstance(True, int)` is `True`). When Developer agents generate overly complex checks such as `if not isinstance(n, int) or isinstance(n, bool):`, it introduces cognitive load and fragile logic. Utilizing an elite Reviewer agent (Gemini 2.5 Pro) to reject this code and route strict diagnostics forces the Developer agent (Claude) to align with a precise type-identity pattern: `if type(n) is not int:`.

* **Sandbox Volume Isolation & Anti-Self-Mutation** *([ADR 0004](./docs/decisions/0004-modularization-sandbox-hardening.md))*: Mounting the entire working directory (`cwd`) inside execution containers exposes the pipeline's own core files to accidental or malicious modifications by run-away agent scripts. Transitioning to a strict dual-mount strategy—mounting framework code `src/` as Read-Only (`:ro`) and the volatile execution output `artifacts/` as Read-Write (`:rw`)—completely mitigates the risk of self-mutation.

* **Modularization Mandate** *([ADR 0004](./docs/decisions/0004-modularization-sandbox-hardening.md))*: Monolithic orchestration (500+ lines in `orchestrator.py`) has become a primary bottleneck for maintainability and security auditing. Transitioning to a decoupled, module-based architecture (Logic/Nodes/Utils) is mandatory to mitigate path-traversal risks and allow atomic testing of pipeline components.

* **Observability Bottlenecks** *([ADR 0003](./docs/decisions/0003-dual-channel-observability.md))*: Standard output (`print()`) is highly volatile and insufficient for autonomous trace tracking. A strict dual-channel logging system (Console INFO + Rotating Audit Log DEBUG) is required to ensure complete pipeline repeatability. Token counts must be extracted natively from structured responses, while out-of-band CLI executions (Claude CLI) must be audited through specialized session database parsers (`npx ccusage`).

* **LTS vs. Experimental Model Quotas** *([ADR 0003](./docs/decisions/0003-dual-channel-observability.md))*: Pinning cutting-edge experimental preview models to Free Tier accounts introduces severe API bottlenecks (429 Resource Exhausted under a 20 request/day cap). For PoC development, shifting workloads to robust mainstream versions (Gemini 2.5 Flash and Pro) balances logical reasoning capacities with highly generous execution quotas.

* **Implicit Type Hazards** *([ADR 0002](./docs/decisions/0002-async-qa-node-isolation.md))*: LLM reasoning consistently misses language-specific type inheritance constraints. Multi-agent code generation requires explicit runtime type-guards to counter architectural assumptions.

* **QA Boundary Isolation** *([ADR 0001](./docs/decisions/0001-baseline-sequential-loop.md))*: Never grant the Developer agent write permissions to the validation scope. Without absolute environment isolation, models will mutate tests to fake a successful compilation.

* **Hybrid Model Routing & Caching** *([ADR 0000](./docs/decisions/0000-cloud-infra-fsm-research.md))*: Multi-agent architectures are cost-prohibitive without strict model routing driven by tool access. Ingest heavy context via Gemini Context Caching, and bind active code mutations to Claude Code execution using ephemeral Prompt Caching to achieve up to a 90% input token discount.

* **The Custom Framework Mandate** *([ADR 0000](./docs/decisions/0000-cloud-infra-fsm-research.md))*: Generic graph frameworks (e.g., LangGraph) introduce heavy dependency bloat and rigid boilerplate. Building a custom Python/Pydantic Finite State Machine (FSM) is required for absolute latency control, programmatic circuit breaking, and native visibility into execution loops.
