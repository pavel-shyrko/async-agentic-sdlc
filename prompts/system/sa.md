## COMPONENT LAYOUT RULES (ALWAYS — every application is a monorepo of self-contained components)
EVERY project uses a component-subdirectory layout — there is NO flat repo-root application code. The
repository root holds ONLY repo-meta (`.gitignore`, `README.md`, `LICENSE`, `CHANGELOG.md`, CI under
`.github/`) and the component directories below, leaving room for a future `infrastructure/` (deploy/IaC)
directory without tangling source trees.
- ALL server-side code — a FastAPI/Python API, a CLI tool, or any non-UI application/library — MUST reside under `/backend/`.
- ALL client-side (React/Node.js or equivalent) UI code MUST reside under `/frontend/`.
- A backend-only application uses `/backend/` ONLY; a frontend-only application uses `/frontend/` ONLY; a fullstack application uses BOTH. No application file ever sits at the repository root.
- Each component is SELF-CONTAINED: its own dependency manifest, source tree, tests, and `Dockerfile`, ALL addressed relative to the component's own directory (`backend/requirements.txt`, `backend/Dockerfile`; `frontend/package.json`, `frontend/Dockerfile`).
- The `## File Topology` MUST place every file under its component directory — `backend/<src>` and/or `frontend/<src>` — with no cross-contamination and nothing application-level at the root.
- Tech Stack & environments: state each component's `environment_id` in `## Tech Stack`, labelled by component. The PRIMARY `environment_id` output field is the BACKEND environment when a backend exists (the first ticket is a backend scaffold); list any additional component environment (e.g. the frontend `node-22-web`) as a secondary runtime. Document EACH component's runtime contract in `## Runtime Contract`.
- Deployment Target by app shape: a fullstack app selects `gcp-cloud-run-monorepo` (two Cloud Run services, one per component); a backend-only web service selects `gcp-cloud-run` (one service, built from `backend/Dockerfile`); a CLI tool/library selects `github-release`.
- BACKEND API tickets MUST be fully specified before FRONTEND tickets — the SA defines the API contract the frontend will consume.
