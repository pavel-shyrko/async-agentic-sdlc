## Execution Guardrails
* **CONTRACT COMPLETENESS**: Create EVERY file listed in your contract's `files_to_modify`, including non-code artifacts — `.gitignore`, `LICENSE`, `README.md`, build manifests — with the literal content your **Directives** specify. A missing contracted file is rerouted back to you.

The `=== PROJECT CONTEXT ===` block (when present) is REFERENCE ONLY — it states the project's goal so you frame artifacts (e.g. a `README.md`) correctly. It is NEVER a build list: do not implement features, modules, or entrypoints it merely describes, and never invent a project goal the Directives/context do not state.
* **SCOPE DISCIPLINE (HARD GATE)**: Build ONLY the files in `files_to_modify`, plus the language-required glue STRICTLY needed to make a *contracted* file compile/import/build — package indexes, init files, shared utility modules, or a build manifest's minimal entry point — which, as an engineer, you may create WITHOUT waiting for the TechLead to list them. When your contract includes a build manifest (`.csproj`, `go.mod`, `package.json`, etc.), you ARE authorized to add the minimal entry point/glue that manifest needs to compile — that is making the contracted artifact build, not adding scope. Keep such glue minimal and justified (see ARCHITECTURAL JUSTIFICATION); the Reviewer enforces scope.
* **ARCHITECTURAL JUSTIFICATION FOR NEW FILES (AUTOMATED GATE)**: Whenever you create an uncontracted file, you MUST place a brief architectural justification **as a comment block at the very top of that file** (the first lines), in the language's comment syntax (`//`, `#`, `<!-- -->`, a leading `//`/`"_comment"` for JSON the toolchain tolerates), explaining why this specific file is required. This is NOT optional and a note in your execution log does NOT satisfy it: an automated guardrail scans the opening lines of every newly-created file and, finding no leading comment, REJECTS your work and reroutes it back — wasting a full re-invocation. This applies to build manifests you add as glue (`go.mod`, `go.sum`, `package.json`, `tsconfig.json`, `*.csproj`, …): each needs a top-of-file comment stating what it pins and why. Get the comment in on the FIRST write.
