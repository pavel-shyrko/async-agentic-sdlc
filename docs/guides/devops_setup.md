# DevOps Setup — GitHub Organization → GCP WIF (Passwordless CI/CD)

One-time platform setup that makes the manifests generated by the **`--scaffold-deploy`** phase (E4, the
DevOps agent) actually deploy. It wires a **passwordless** trust bridge between a GitHub **organization** and
Google Cloud so every repository in the org inherits the same deploy pipeline — no long-lived service-account
keys anywhere. Do this **once** per organization; afterwards any new repo deploys with zero extra setup.

> Scope: this guide configures the *target* (the org + GCP). The engine only **generates and merges** the
> workflow + Dockerfile ([prompts/skills/devops_*.md](../../prompts/skills/), [devops.py](../../src/deployment/agents/devops.py));
> GitHub Actions performs the WIF deploy on the next push. The engine never holds GCP credentials.

## Integration passport

- **Protocol:** OpenID Connect (OIDC) via Workload Identity Federation (WIF).
- **Security principle:** No-Credentials Policy — no long-lived service-account JSON keys in the system.
- **Trust scope:** GitHub **organization** level (`attribute.repository_owner`). Every new repo in the org
  inherits the deploy pipeline automatically.
- **Target GCP region:** `europe-central2` (Warsaw, Poland).
- **Artifact registry:** a shared pool `software-factory` in Google Artifact Registry.

## Step 1 — GCP cloud infrastructure

Run these in Google Cloud Shell, in order.

### 1.1 Initialize environment variables and pin the context

```bash
export PROJECT_ID="cyberthon-2026-token-burners"
export REGION="europe-central2"
export GITHUB_ORG="Cyberthon-2026-Token-Burners"
export REGISTRY_NAME="software-factory"

# Pin the active project
gcloud config set project $PROJECT_ID
```

### 1.2 Enable the dependent Google APIs

```bash
gcloud services enable iam.googleapis.com \
    iamcredentials.googleapis.com \
    artifactregistry.googleapis.com \
    run.googleapis.com \
    cloudbuild.googleapis.com
```

> **Note:** `run.googleapis.com` is required for Cloud Run deployments. Even with correct IAM
> bindings, `gcloud run deploy` returns `PERMISSION_DENIED: SERVICE_DISABLED` if this API is not
> enabled. `cloudbuild.googleapis.com` is often co-required by Cloud Run's internal build pipeline.

### 1.3 Create the shared Artifact Registry

```bash
gcloud artifacts repositories create $REGISTRY_NAME \
    --repository-format=docker \
    --location=$REGION \
    --description="Shared Software Factory Container Registry"
```

### 1.4 Create the service account and grant registry rights

An isolated identity for CI/CD pushes, authorized to write to the registry:

```bash
# Create the SA
gcloud iam service-accounts create sa-github-devops \
    --display-name="Automated GitHub DevOps Agent SA"

# Grant Artifact Registry write access
gcloud projects add-iam-policy-binding $PROJECT_ID \
    --member="serviceAccount:sa-github-devops@$PROJECT_ID.iam.gserviceaccount.com" \
    --role="roles/artifactregistry.writer"

# Grant Cloud Run deploy rights (required to create/update Cloud Run services)
gcloud projects add-iam-policy-binding $PROJECT_ID \
    --member="serviceAccount:sa-github-devops@$PROJECT_ID.iam.gserviceaccount.com" \
    --role="roles/run.admin"

# Grant permission to act as the runtime service account (required by Cloud Run)
gcloud projects add-iam-policy-binding $PROJECT_ID \
    --member="serviceAccount:sa-github-devops@$PROJECT_ID.iam.gserviceaccount.com" \
    --role="roles/iam.serviceAccountUser"
```

### 1.5 Create the Workload Identity pool and OIDC provider (with an attribute condition)

The hardened `--attribute-condition` rejects any request from outside the organization; the attribute
mapping fixes GitHub's standard subject claim (`assertion.sub`) and surfaces `repository_owner`:

```bash
# 1. Create the Workload Identity pool
gcloud iam workload-identity-pools create factory-pool \
    --location="global" \
    --display-name="Factory Pool"

# 2. Create the OIDC provider with owner mapping + claim validation
gcloud iam workload-identity-pools providers create-oidc factory-provider \
    --workload-identity-pool="factory-pool" \
    --location="global" \
    --display-name="Factory Scale Provider" \
    --attribute-mapping="google.subject=assertion.sub,attribute.repository=assertion.repository,attribute.repository_owner=assertion.repository_owner,attribute.actor=assertion.actor" \
    --attribute-condition="assertion.repository_owner == '$GITHUB_ORG'" \
    --issuer-uri="https://token.actions.githubusercontent.com"
```

> **Case sensitivity:** the `repository_owner` claim is matched **exactly**. Keep `$GITHUB_ORG` in its
> canonical GitHub login casing or the condition silently denies every token.

### 1.6 Bind the org-wide pool policy to the service account

Allow the pool to impersonate the SA for **every** repository owned by the org:

```bash
# Resolve the pool's full resource name
export POOL_NAME=$(gcloud iam workload-identity-pools describe factory-pool \
    --location="global" --format="value(name)")

# Bind the SA to the org-owner principalSet
gcloud iam service-accounts add-iam-policy-binding \
    sa-github-devops@$PROJECT_ID.iam.gserviceaccount.com \
    --role="roles/iam.workloadIdentityUser" \
    --member="principalSet://iam.googleapis.com/${POOL_NAME}/attribute.repository_owner/$GITHUB_ORG"
```

### 1.7 Export the pipeline metadata

Print the exact identity-provider string the workflow will reference:

```bash
gcloud iam workload-identity-pools providers describe factory-provider \
    --workload-identity-pool="factory-pool" \
    --location="global" \
    --format="value(name)"
```

*Record the output (the **WIF Provider String**) and the **service-account email** for Step 2.*

## Step 2 — GitHub organization configuration

For org-wide inheritance, go to `Organization Settings ➔ Secrets and variables ➔ Actions`.

### 2.1 Organization secrets

Add under *Secrets ➔ New organization secret*, with **Repository access: All repositories**:

| Secret | Value |
|---|---|
| `GCP_WIF_PROVIDER` | the provider string from 1.7, e.g. `projects/<PROJECT_NUMBER>/locations/global/workloadIdentityPools/factory-pool/providers/factory-provider` |
| `GCP_SERVICE_ACCOUNT` | `sa-github-devops@cyberthon-2026-token-burners.iam.gserviceaccount.com` |

### 2.2 Organization variables

Add under *Variables ➔ New organization variable*, with **Repository access: All repositories**:

| Variable | Value |
|---|---|
| `GCP_PROJECT_ID` | `cyberthon-2026-token-burners` |
| `GCP_REGION` | `europe-central2` |
| `GCP_REGISTRY_NAME` | `software-factory` |

### 2.3 HARD GATE — workflow permissions

GitHub blocks OIDC for new organizations by default. You **must** flip this:

1. `Settings ➔ Actions ➔ General` (org or repo).
2. Scroll to **Workflow permissions**.
3. Set **Read and write permissions**.

Without it, the generated YAML's `permissions: id-token: write` fails with `Permission Denied`.

### 2.4 Allow `github-actions` to bypass the default-branch protection (post-deploy README stamp)

The deploy/release workflow's final step stamps the live deployment / release URL into `README.md` and
pushes it to the default branch (`git push origin HEAD:<default-branch>`, the detached-HEAD-safe refspec —
see the `deploy_*` platform skills). If the default branch is **protected**, that push is rejected unless the
robot identity is allowed to bypass the rule. Grant it once:

1. `Settings ➔ Branches` (org ruleset or per-repo branch protection rule for the default branch).
2. Under **"Allow specified actors to bypass required pull requests"** (rulesets) / **"Allow bypass"**, add
   the **`github-actions`** app (or your service account / token identity).

Without this, the URL-stamp push fails with `protected branch hook declined` (the build/deploy itself still
succeeds; only the cosmetic README update is skipped). The push uses the default `GITHUB_TOKEN` and carries
`[skip ci]`, so it never loops. If you prefer not to relax protection, simply drop the README-URL step — the
live URL is still available from the deploy step's logs and the GitHub Release page.

## Step 3 — Autonomous operation (zero-human-touch)

Once Steps 1–2 are done the bridge is permanent. To ship any new app:

1. Create an empty repository in the org (e.g. `json-csv-converter`).
2. Run the pipeline with `--scaffold-deploy`:

```bash
python3 main.py \
  --idea "CLI JSON to CSV Converter. Transforms nested JSON arrays into RFC 4180 CSV via dot notation. Configurable flags -d/--delimiter." \
  --repo "git@github.com:Cyberthon-2026-Token-Burners/json-csv-converter.git" \
  --auto-execute \
  --scaffold-deploy
```

### Autonomous trace

- The engine develops the code and merges each feature into `main` (E3 `run_batch`).
- In the terminal phase the DevOps agent classifies the archetype (`cli_tool`), pulls the
  [devops_cli_tool.md](../../prompts/skills/devops_cli_tool.md) skill, suppresses the `Dockerfile`, and emits
  a valid build/publish workflow.
- The manifest passes `run_devops_gate` (static YAML + directive lint), is pushed via the
  `chore/devops-scaffold` PR, and squash-merges into `main`.
- Native GitHub Actions picks up the workflow, requests a short-lived GCP token over WIF, and publishes the
  ready-to-run artifact/image to the Warsaw registry.

## Engine contract — what the generated workflow references

The deploy mechanics live in the registry-driven platform skills
([prompts/skills/deploy_gcp.md](../../prompts/skills/deploy_gcp.md) for Cloud Run,
[prompts/skills/deploy_github_release.md](../../prompts/skills/deploy_github_release.md) for a CLI), kept
separate from the app-shape archetype skills (`prompts/skills/devops_*.md`). The DevOps agent must emit a
workflow that reads:

- **Secrets:** `${{ secrets.GCP_WIF_PROVIDER }}`, `${{ secrets.GCP_SERVICE_ACCOUNT }}`
- **Variables:** `${{ vars.GCP_PROJECT_ID }}`, `${{ vars.GCP_REGION }}`, `${{ vars.GCP_REGISTRY_NAME }}`
- **Permissions:** `id-token: write` + `contents: write` (WIF needs the OIDC token; `contents: write` covers
  the post-deploy README-URL commit, pushed via the `HEAD:<default-branch>` refspec — needs the §2.4 bypass
  on a protected branch).
- **Auth action:** `google-github-actions/auth` (WIF), then `deploy-cloudrun` (web service) or an
  Artifact-Registry / `gh-release` publish step (CLI tool).
- **Public invocation (web service):** the deploy step passes `flags: '--allow-unauthenticated'` so the
  Cloud Run service is reachable anonymously — without it the live service returns **HTTP 403** ("The
  request was not authenticated") for every request. `run_devops_gate` enforces this at scaffold time.
  > **Caveat:** if your org enforces the `iam.allowedPolicyMemberDomains` (domain-restricted sharing) org
  > policy, granting `allUsers` is blocked and the deploy step fails — either exempt the project from that
  > policy or front the service with an authenticated ingress; this is an org-policy decision, not an
  > engine bug.
